What is FIO?

FIO (Financial Intake & Orchestrator) is the central inbox for every invoice, receipt, and bank statement at Amitours Holding. The AI parses each document, validates the VAT number in the official EU registry, classifies it into our chart of accounts, and posts it to the correct Profit Center's P&L.

Document lifecycle

  1. pending — just uploaded, parser is running.
  2. parsed — AI extracted text/structure, awaits classification.
  3. classified — ledger code proposed, ready for human review.
  4. approved — human picked PC + code, but not yet posted to ledger.
  5. posted — landed in the P&L. Visible in Accounting and Analytics.
  6. rejected — declined with a reason, won't appear in P&L.

💾 Save vs ✅ Approve & Post

  • Save changes — persist your edits (Profit Center, Department, Cost Reason, Ledger code, Approver name) without finalizing. The document stays in classified status. Useful when you need to pause halfway, or save partial info for someone else to finish.
  • Approve & Post — commits to ledger. The document is locked, P&L is updated, and Analytics picks it up.
  • Once posted, you can only delete — deletion rolls back the posting from actuals.

Split allocation

  • One invoice can be split across multiple business units when the paying entity (e.g., Alps2Alps OÜ) fronts a cost that benefits another stream (e.g., Skibookers).
  • Open the modal → section Split this invoice across multiple business units → add rows.
  • Sum of Share % must equal exactly 100. Amount column auto-fills as you pick percent (and vice versa).
  • Each row also picks its own ledger code — e.g., MT 30% under CNS0, SR 70% under MSO0.
  • After Approve & Post, each PC's P&L is updated with its slice, and the Upload table shows stacked PC tiles.

VAT & VIES verification

  • The parser extracts VAT numbers from 27 EU countries (LV, EE, LT, DE, FR, etc.) plus Latvian Reg. Nr.
  • If a VAT is found, FIO automatically calls the official EU VIES registry to verify the number and pull the official legal name + address.
  • ✓ green badge — verified, official details auto-filled.
  • × red badge — VAT format looks right but VIES says it's invalid. Possible reasons: vendor recently changed VAT, VIES is temporarily down for that country, or the OCR mis-read a digit. Use the remediation buttons in the document modal.
  • No badge — vendor has no EU VAT (e.g., US company like Fireflies). This is correct, just review manually.

🔗 How FIO talks to BT4YOU Executive Bot

FIO does not invent its own roster of people or department names. It pulls them from BT4YOU Executive Bot — the holding's source-of-truth for who works on what.

  • People (52 humans) come from BT4YOU/data/holding_config.json → nodes[].person_name. Each node also carries the Asana GID, role title, and parent — so when you pick "Uploaded by" in Upload or "Approver" in Approve, you're picking the same Artjoms / Rita / Anastasiia who lives in Asana.
  • Departments (6 of them: Product / BD / F&A / Operations / Marketing / HR) come from BT4YOU/data/brand.json → departments. The colors of the badges and dropdowns are the brand colors defined there.
  • Profit-center ↔ department mapping lives in FIO's services/bt4you_sync.py: Alps2Alps and Holding expose all 6 departments, Skibookers / Skipasser show 4, Mountly shows 3, etc.
  • Why this matters: when Asana adds a new manager to a stream, BT4YOU picks them up first; FIO reads BT4YOU's snapshot on every API call to /api/people. One add — appears everywhere. No duplicate maintenance.
  • Outbound: P&L feed. FIO can push its accounting actuals back to BT4YOU's holding dashboard via /api/sync-bt4you → BT4YOU's /api/holding/cashflow/fact/generic. So the executive bot's CEO view of cashflow comes from FIO posts.
  • Cloud snapshot: the deployed instance on Fly bundles a snapshot of BT4YOU data taken at deploy time (no shared filesystem with your laptop). When BT4YOU changes meaningfully, ping me to re-snapshot.

📄 Anatomy of a document card (View modal)

When you click View on any row, this modal opens. Top to bottom:

  1. Parser-failure banner (red, only if shown) — Anthropic credits ran out, or the file is blank / unreadable. Vendor and amount will be empty; fill them manually.
  2. VIES verification badge:
    • ✓ green — official EU registry confirms the VAT and we auto-filled the legal name + address.
    • × red — VAT looks like one but VIES says invalid. Use the remediation buttons (Re-check VIES / Lursoft for LV / Äriregister for EE / Mark as manually verified with a registry URL).
    • No badge — vendor has no EU VAT (e.g. US company). That's fine, just review manually.
  3. Header card: vendor name, VAT, address, total amount in € + tax/net breakdown, document date. All extracted by Claude Vision.
  4. Terminal-receipt link (only if applicable) — when the parser detected a card terminal receipt that matches a purchase invoice, both are linked here.
  5. Expense Policy badges — quick green/yellow/red status checks (corporate-policy compliance, missing VAT, etc.).
  6. Warnings — anything the parser wanted to flag (e.g. "currency ambiguous", "multiple totals detected").
  7. Proposed Classification — Claude's top 2-3 ledger codes with confidence scores. Click any to select. Codes ≥ 90% confidence auto-post on upload.
  8. Override — manual ledger code + profit center, in case Claude's suggestions are wrong. Hidden when the doc is Split-allocated (each split row carries its own code).
  9. Line Items — each detected line of the invoice. If the invoice mixes categories (e.g. food + travel on one bank statement), per-line dropdowns let you attribute each line separately. Hidden when split.
  10. Cost allocation (collapsed block) — this is the "Split" feature. Add rows of Profit center / Ledger code / Share % / Amount € / Note. Picking percent auto-fills amount and vice-versa. Sum of shares must be 100. Save allocation commits the split — if the doc was already posted, FIO automatically subtracts the old single-PC posting and re-posts to each stream pro-rata (no double-counting).
  11. Bottom controls (action bar):
    • Profit Center / Ledger code / Department / Cost reason / Approver — five labelled dropdowns.
    • 💾 Save changes (blue) — persist edits without posting. Button pulses orange when there are unsaved changes.
    • ✅ Approve & Post (green) — finalize. Writes to P&L, appears in Accounting and Analytics.
    • 🚫 Reject (red) — decline with a reason.
    • "Always classify this vendor as selected code" — train the classifier. Next document from the same vendor auto-picks this code.
  12. 👍 / 👎 feedback buttons (in the Approve table row, not in the modal) — quick ML feedback for the classifier. 👎 opens a what's-wrong checklist.

🚨 Suspicious findings (Analytics)

  • 🔴 high: amount outlier (>3 sigma above mean for the same code).
  • 🟠 medium: vendor charged under 3+ codes (likely mis-tagged) · posted document without VAT · fuzzy duplicate vendor names.
  • ⚪ low: foreign (non-Baltic) vendor · round-number amounts ≥ €500.

Tips & gotchas

  • The app sleeps after ~10 min idle on the free Fly tier. First request after sleep takes 1–3 seconds to wake.
  • If the UI looks stale (no Save button, missing dropdown) hit Cmd + Shift + R to bust the browser cache.
  • Upload size limit: 20 MB per file. Larger PDFs — compress or split first.
  • The AI parser depends on a paid API. When credits run out, a red banner explains it; regex & VIES still try to extract VAT/address, but vendor name & amounts must be filled manually.
  • Pre-prod tool, no auth yet. Don't share the URL outside the company.
